### TRACK C — Security, Privacy and Assurance

*What an attacker, a regulator, or an acquirer's counsel will ask — and what you will wish you had written down, given that nobody wrote anything down.*

**C-01 · Core application security** — Priority **10/10** — `STOP-SHIP` — *`S3`*
- **Probe:** Attack it. Authenticate as user B and request an object belonging to user A by changing the identifier in the URL — do this for **every** endpoint that accepts an object identifier, not one. Test authentication, session handling, injection, server-side request forgery, cryptography and secret handling. Check whether any authorisation decision is made in the browser. This remains where the overwhelming majority of real incidents land; AI features sit on top of this baseline, they do not replace it.
- **Passes only if:** the application is verified against a defined application-security verification level, every authorisation decision is enforced server-side, and there are no unresolved high or critical findings.
- **Targets:** Level 2 of the application-security verification standard for the whole application (Level 3 for anything identity-related); **100% of authorisation decisions server-side and covered by automated tests** — those tests are the *only* thing standing between this system and a cross-tenant breach, because no reviewer will ever look at the query again; static, dynamic and composition analysis in CI blocking on high/critical; every endpoint mapped against the API-security risk list; all secrets in a vault, none in source, scanned on every commit.
- **On failure:** stop. Fix the access-control defects before anything else in this document. Then sweep — **a broken ownership check is never one route, it is a template** (§0, hazard 3), and the template was applied by something that does not get bored.
- **Structural fix (S13):** **Push tenancy into the data-access layer: a query that cannot be constructed without an ownership predicate.** Not a check added to fifteen routes — a type, a repository interface, or a session-scoped query builder that makes the cross-tenant query unrepresentable. **You stop finding this bug because you can no longer write it**, and the clone class behind it (`A-07`) dies with it. Fix the sixteenth route by hand and you will be back here for the seventeenth.
- **Standing control:** *A route without an authz test cannot merge.* The authorisation-coverage gate runs on every build; a DAST authz sweep runs nightly against a running environment. **Ratchet (S11):** routes without an authz test remain at zero; high/critical findings remain at zero. **Calibration (S12):** a cross-tenant defect is seeded weekly and the gate must catch it — **these tests are the only thing standing between this system and a cross-tenant breach, so they are themselves tested.** **Blocks:** merge and deploy. **Owner:** platform-security.

**C-02 · Threat model** — Priority **8/10** — *`S1` `S3`*
- **Probe:** Ask for the threat model. If it does not exist, build it from the system as it stands: data-flow diagram, trust boundaries, and a structured enumeration (spoofing, tampering, repudiation, information disclosure, denial of service, elevation of privilege) at each boundary. **Then ask the question this operating model forces: what keeps it current?** A threat model maintained by a human who reviews architecture changes is a threat model that will be current for exactly as long as that human exists — which here is zero days.
- **Passes only if:** every significant feature has a maintained threat model with explicit trust boundaries, every identified threat traces to a control, a test, or an accepted-risk record with a compensating control and a tripwire — **and the model is kept current mechanically rather than by attention.**
- **Targets:** a data-flow diagram with trust boundaries per service; a structured threat analysis per boundary; the model in version control; **a CI check that fails when a new trust boundary, external integration, tool or egress path appears without a corresponding threat entry** — that check is the substitution for the architect who would have noticed the new integration in review; **every threat mapped to a control, a test, or a documented acceptance — no orphans**; a design gate before launch, enforced by policy.
- **On failure:** build it, then wire the staleness check. **Design-time analysis catches the architectural flaws no scanner will ever find** — and in this system, nothing else performs design-time analysis at all.
- **Structural fix (S13):** Generate the architecture from the running system and diff the threat model against it, rather than maintaining a diagram that was accurate once (§6.5.3). See `A-28` `C-09` `C-16`.
- **Standing control:** *New-trust-boundary detector.* CI fails when a new integration, tool, egress path or data class appears without a corresponding threat-model entry; the model is diffed nightly against the architecture generated from the code. **This is the substitution for the architect who would have noticed the new integration in review** — and it is what keeps the threat model from being accurate exactly once. **Ratchet (S11):** orphan threats and unmapped boundaries remain at zero. **Blocks:** merge. **Owner:** ai-security.

**C-03 · Supply-chain security and hallucinated packages** — Priority **9/10** — *`S1`*
- **Probe:** See B-04, and go further: check whether any dependency was *first registered* shortly after this code was generated. **Attackers profile the package names models reliably invent and pre-register them** — roughly 43% of hallucinated package names recur across repeated generations, which makes them a stable, targetable set. This is the attack designed specifically for a codebase like this one.
- **Passes only if:** no dependency enters a build without provenance and existence verification, and hallucinated, unpinned or newly-registered packages **cannot install**.
- **Targets:** all dependencies pinned by hash or lockfile; a package-existence and allowlist check in CI, blocking; composition analysis on every build; blocked installation of newly-registered or low-reputation packages; **a dependency-verification step applied specifically to coding-assistant output, run before the package is ever installed** — not after, and not by a person reading the diff; a written playbook for the day you find a poisoned dependency.
- **On failure:** verify every package mechanically. **This is the attack path most specific to a machine-built codebase, it is being actively exploited, and it is the single clearest example of what is lost when nobody reads the import block.**
- **Standing control:** *Registry age and reputation re-checked nightly on the full lockfile.* As B-04, plus a rehearsed poisoned-dependency playbook. **Ratchet (S11):** unverified dependencies remain at zero. **Calibration (S12):** a plausible-but-nonexistent package name is seeded monthly and must be blocked before install. **Blocks:** build. **Owner:** platform-security.

**C-04 · Privacy and data governance** — Priority **10/10** — `STOP-SHIP` *(where EU personal data is processed; otherwise re-band against the law that applies and name it)*
- **Probe:** Follow the personal data. Where does it enter, where is it stored, where is it copied (logs, traces, caches, embeddings, analytics, the model provider), how long does it stay, and on what lawful basis? Then **test a deletion request end-to-end** and see whether it actually works.
- **Passes only if:** all personal-data processing has a documented lawful basis, a signed impact assessment where required, and **technically enforced** minimisation and retention.
- **Targets:** an impact assessment completed and signed **before** processing began (this is a legal artifact authored in-command, not a code review — it survives the removal of the reviewer intact); a current data map / record of processing; retention limits with **automated deletion**; encryption at rest and in transit; consent and subject-access workflows **tested end to end automatically and on a schedule**, because a deletion path exercised only by hand will rot and you will learn about it from a regulator; personal-data access least-privilege and logged. Exposure is up to €20M or 4% of global turnover — and this is the first thing an acquirer's counsel asks for.
- **On failure:** stop processing without a lawful basis. Everything else here is engineering; **this is law, and law does not care about your operating model.**
- **Structural fix (S13):** The derived-store registry (see `B-37`), plus a data map generated from code and infrastructure. The record of processing stops being a document somebody must remember to update — **and there is nobody to remember.**
- **Standing control:** *Retention enforced automatically; the data map generated, not written.* The map is regenerated from code and infrastructure and diffed against the record of processing — **a new store that touches personal data and is not in the map fails the build.** The erasure and subject-access workflows are tested end-to-end on a schedule against a canary subject (B-37). The impact assessment is re-triggered automatically when a new data class or purpose appears. **Ratchet (S11):** unmapped personal-data stores remain at zero; retention windows may shorten, not lengthen. **Blocks:** deploy. **Owner:** privacy (in-command; the lawful basis is a human judgement and stays one).

**C-05 · The LLM application risk taxonomy, mapped and tested** — Priority **9/10** — *`S3`*
- **Probe:** Build a coverage matrix. Ten categories: prompt injection; sensitive-information disclosure; supply chain; data and model poisoning; improper output handling; excessive agency; system-prompt leakage; vector and embedding weaknesses; misinformation; unbounded consumption. For each: which control addresses it here, and **which test proves the control works?**
- **Passes only if:** every category has a documented position, a control, and a test — **with non-applicability justified in writing, never assumed.**
- **Targets:** a matrix mapping all ten categories to a control **plus an executable test** (a control with no test is, in this system, a claim); injection and output-handling tests in CI; consumption caps; a sensitive-disclosure evaluation; vector-store access controls; release blocked on any regression. Note that sensitive-information disclosure has risen to the second position in this taxonomy and excessive agency has been substantially expanded — **the risk has shifted toward what the system *reveals* and what it is *allowed to do*, not merely what it *says*.**
- **On failure:** the matrix is the deliverable. **An empty cell is a finding, and a cell containing a control but no test is a finding wearing a disguise.**
- **Standing control:** *Coverage matrix with a test per cell, enforced.* CI fails on an empty cell. **Cadence:** the matrix is re-validated when the taxonomy publishes a new edition, and annually regardless. **Ratchet (S11):** cells with a control but no test may only decrease — **a control with no test is a claim, and claims are what this system produces most fluently.** **Blocks:** release. **Owner:** ai-security.

**C-06 · The agentic risk taxonomy** — Priority **8/10** — *`S5` `S6` — becomes priority 10 if any model here can call a tool that writes, sends, spends, deletes or executes*
- **Probe:** Ten categories, all of which only exist once a system plans and acts: goal hijack; tool misuse; identity and privilege abuse; agentic supply chain; unexpected code execution; memory and context poisoning; insecure inter-agent communication; cascading failures; human-agent trust exploitation; rogue agents. Walk each against this architecture. **Note that one of them — human-agent trust exploitation — mutates in this operating model rather than disappearing: with no human in the loop, the exploited trust is no longer a person's trust in the agent. It is the *pipeline's* trust in the agent's output.** Audit it in that form.
- **Passes only if:** the agentic system is audited against all ten, with least agency and execution observability enforced as **architectural properties**, not as policies.
- **Targets:** a per-agent capability inventory; least-agency scoping documented; full execution traces (goal state, every tool call, every argument) logged and queryable; a tested kill switch that **fires itself** (A-34); **every irreversible action removed from the capability set or wrapped in dry-run → deterministic precondition validation → apply → post-condition verify → automatic compensation (S5)** — "a human approves it" is not an available control here, and an irreversible capability with neither a validator nor a compensating transaction is a `STOP-SHIP` on its own; a coverage matrix with tests. These are the risks that static application controls miss entirely, and that even the non-agentic LLM taxonomy does not reach.
- **On failure:** if you cannot answer **"what is this agent allowed to do, what is the worst it can do, and what stops it by itself,"** you do not have an agentic system under control. You have one running.
- **Structural fix (S13):** The tool gateway (see `A-11`) — capability labels are declared at registration and the gateway refuses an unlabelled tool, so the capability inventory is generated rather than maintained.
- **Standing control:** *The capability inventory is generated from the running configuration and diffed.* An agent capability added without an agentic-risk mapping fails the build. **Ratchet (S11):** irreversible capabilities without a deterministic validator and a compensating transaction remain at zero — permanently, and a breach is a `STOP-SHIP` (§3). **Calibration (S12):** an unmapped capability is introduced on a scratch branch monthly and must be caught. **Blocks:** merge and deploy. **Owner:** ai-security.

**C-07 · Prompt injection** — Priority **9/10** — **UNSETTLED DOCTRINE** — *`S5`*
- **Probe:** Attempt direct injection (in user input) and indirect injection (in anything the system retrieves, reads or is sent). Run a recognised injection benchmark suite against the deployed system. **Then read this and internalise it: there is no reliable general defence.** A 2025 study by researchers from three frontier labs bypassed twelve recently published defences with attack success rates above 90% for most of them — defences whose original papers had reported near-zero success rates. A human red-teaming competition with 500 participants achieved a 100% success rate. **Treat any vendor claim of a solved defence as unverified marketing.**
- **Passes only if:** containment is **architectural** — a successful injection cannot trigger a consequential action — and the residual risk is measured, explicitly accepted in writing, and carries a compensating control and a runtime tripwire.
- **Targets:** all model input treated as untrusted; a *deterministic* boundary (capability-based control, a dual-model split, or taint-tracking plus a policy gate) rather than filters alone; an injection evaluation suite in CI, blocking on regression; a written acknowledgement that no defence is complete; runtime monitoring **with automatic containment** (B-20).
- **On failure:** **do not write a classifier and mark this closed.** That is the single most common false remediation in this field. **And do not propose a human approver as the compensating control — that is the second most common one, and here it is not merely weak but fictional.** Remove the capability that makes a successful injection consequential. That is the only control that has ever held.
- **Structural fix (S13):** Split the agents (see `B-20`): untrusted text never shares a session with a tool that can act or a channel that can egress. **The residual risk on this check is UNSETTLED and will stay that way. The only durable mitigation is architectural, not detective** — you will not filter your way out of this one.
- **Standing control:** *Injection suite blocking, corpus refreshed monthly, containment asserted at runtime.* **A frozen injection corpus is the classic silent failure here: it will keep passing forever while proving only that you can defeat the attacks of the year you built it.** The residual risk is re-stated and re-accepted on a cadence, and its tripwire is verified rather than assumed. **Ratchet (S11):** attack-success rate may not rise above the recorded ceiling. **Blocks:** release. **Owner:** ai-security.

**C-08 · The dangerous three: never combine all of them** — Priority **8/10** — *`S5` `S6`*
- **Probe:** Label every capability in every agent session with which of the three legs it introduces: **[untrusted input]**, **[private data]**, **[external communication]**. Then find every session that holds all three. That set is your exposure — and it is **checkable**, which is precisely why it matters so much here: unlike prompt-injection defence, this rule is *decidable by inspection*, and a machine can decide it. In a system with nobody to exercise judgement, **the controls that survive are the ones that require none.**
- **Passes only if:** no agent session combines all three legs. Where all three are genuinely unavoidable, **the exfiltration leg is reduced to a deterministic, allowlisted, schema-validated channel** — or the session is refused.
- **Targets:** every capability labelled; **at most two of the three per session, enforced in configuration and asserted by a test**; where all three are genuinely required, the outbound leg is constrained to a deterministic validator — an allowlist of destinations, a schema on the payload, a taint check — **never an approver**; a re-audit triggered **automatically** whenever a tool or integration is added (adding one tool can silently complete the triangle, and no one is watching the tool registry); taint-tracking and a policy gate on every exfiltration-capable action — outbound HTTP, email and chat sends, pull-request creation, rendered links and rendered images.
- **On failure:** **remove a leg.** This is the best practical mitigation available *precisely because* C-07 has no general solution, and it is the one architectural rule in this document that a machine can verify completely, every build, forever.
- **Structural fix (S13):** **The trifecta cannot assemble if the three legs live in different agents with different credentials** (see `B-20`, `A-11`). This is the difference between a check that must be re-evaluated on every tool addition forever, and a property of the deployment topology.
- **Standing control:** *The two-of-three rule asserted as a test, on every build and on every tool addition.* Capability labels are machine-readable and the rule is evaluated mechanically. **This check is the one most likely to be broken silently by a single well-meaning tool addition that completes the triangle**, so the re-audit is automatic and it fails the build rather than warning. **Ratchet (S11):** sessions holding all three legs remain at zero. **Calibration (S12):** a trifecta-completing tool is proposed on a scratch branch monthly and must be blocked. **Blocks:** merge. **Owner:** ai-security.

**C-09 · EU AI Act — and the one place this operating model meets a wall** — Priority **9/10** — *hard gate at 10 if anything here is high-risk on the EU market*
- **Probe:** Inventory every AI system and classify it by risk tier. Then evidence the obligations for that tier. **Verify the current legal position before you rely on any date** — the timeline is in flux and the deferrals below are provisional.
- **Passes only if:** every AI system is inventoried and classified, with the obligations for its tier evidenced.
- **Targets:** a complete AI-system inventory with risk classification; conformity documentation and **human-oversight design** for anything high-risk; transparency and disclosure obligations live for generative features; a documented position on the prohibited practices. **Timeline as last recorded, to be re-verified:** general-purpose model obligations have applied since 2 August 2025; a provisional political agreement of 7 May 2026 defers standalone high-risk obligations to 2 December 2027 (and the product-embedded category to 2 August 2028), **but Article 50 transparency remains at 2 August 2026, with machine-readable marking due 2 December 2026**. If the deferral is not formally adopted, the original dates stand. Exposure is up to €35M or 7% of global turnover.
- **The boundary, and read it carefully.** This mandate removes humans from the **delivery and verification loop**. It does not, and cannot, remove a legal duty of human oversight over a high-risk system's **decisions about people**. These are different things, and conflating them is how an organisation talks itself into an unlawful deployment. Where Article 14 binds, the oversight must be *effective*. It may legitimately be discharged **in-command** — a human authors the policy, holds halt authority, receives the logs, and audits after the fact — rather than **in-the-loop**, where a human approves each decision. The in-command form is not a loophole and is arguably the stronger reading: at this system's decision velocity, per-decision approval degrades into rubber-stamping within weeks, and the automation-complacency evidence on that is unambiguous. **But if the obligation genuinely cannot be met in-command, the correct answer is not to deploy the high-risk system.** It is *never* to reinsert a per-decision approver who will not read what they approve. That would satisfy an auditor and protect nobody.
- **On failure:** classify first. You cannot evidence obligations for a system you have not categorised, and the inventory is a day's work.
- **Structural fix (S13):** Generate the AI-system inventory from the running configuration (§6.5.3). An inventory maintained by hand is wrong by the second release, and nobody here reads the second release.
- **Standing control:** *The AI-system inventory is generated from the running configuration, not maintained by hand.* A new AI system without a risk classification blocks its own deploy; the Article 50 disclosure is asserted by a UI test on every build. **Cadence:** the regulatory-date watch is calendared against an owning role — **in-command, and unavoidably human, because nothing in this pipeline reads the Official Journal.** **Ratchet (S11):** unclassified AI systems remain at zero. **Blocks:** deploy. **Owner:** legal + ai-platform.

**C-10 · Evaluation methodology** — Priority **8/10** — *`S1`*
- **Probe:** Ask what "correct" means for the non-deterministic parts of this system, and **who decided** — because in this operating model, deciding what "correct" means is one of the very few things a human still does, and if nobody did it, then nothing here has a definition of correct at all. Find the golden dataset. Check that it is versioned, that thresholds were set *before* the results came in, and that the pipeline blocks on regression.
- **Passes only if:** an offline and online evaluation regime exists, with versioned golden datasets and pre-agreed thresholds that block release on regression.
- **Targets:** golden/evaluation datasets under version control; a defined acceptance threshold per metric, **agreed in advance and frozen** (a threshold set after seeing the results is not a threshold, it is a rationalisation — and a machine will rationalise faster than you can object); evaluation running in CI with release **blocked** on regression; online evaluation or A/B plus guardrail metrics in production; **evaluation results retained as audit evidence** (S9). Without this there is no regression gate, and a product with no regression gate cannot make a reliability claim anyone should believe.
- **On failure:** build it. **This is the discipline that makes an AI feature shippable at all**, and in a system with no reviewer it is also the only remaining definition of *working*.
- **Standing control:** *Thresholds frozen in version control and enforced by the gate.* **A threshold change requires a decision record, and the ratchet runs one way: thresholds may tighten, never loosen.** A loosening is itself a finding — **this is precisely how evaluation gates rot, one temporary exception at a time, and the exception is never temporary.** Golden-set provenance is recorded and rotation scheduled. **Ratchet (S11):** thresholds monotonically non-decreasing. **Blocks:** release on regression. **Owner:** ai-quality.

**C-11 · A running AI risk-management programme** — Priority **7/10** — *`S9`*
- **Probe:** Look for evidence of a live process — governance, mapping, measurement, management — producing durable artifacts. **A single PDF written once is not a programme**, and a programme whose only output is a PDF is not going to survive an operating model in which nobody reads PDFs.
- **Passes only if:** the programme produces durable evidence artifacts per AI system on an ongoing basis, **generated as a by-product of the pipeline rather than assembled by hand.**
- **Targets:** a governance charter with a named accountable **owning role** (accountability sits in-command; it does not require anyone in the loop); an AI inventory with intended-use documentation; **pre-deployment evaluation results and production monitoring data emitted automatically as evidence artifacts** — if the evidence has to be assembled by a person before an audit, it will be assembled once, and it will be wrong; an incident log and recorded risk treatments; the generative-AI risk profile applied to the LLM features, including prompt injection treated as an information-security risk, which is where it belongs.
- **On failure:** stand up the loop, and make the pipeline emit the evidence. This framework is the de facto benchmark for "reasonable care" in procurement, and it crosswalks to the EU obligations in C-09, so the work is not duplicated.
- **Standing control:** *Evidence emitted by the pipeline, not assembled before an audit.* Each AI system produces its governance, mapping, measurement and management artifacts automatically; a system with no evidence in the current period opens a finding. **Ratchet (S11):** systems with stale evidence remain at zero. **Blocks:** nothing directly; escalates. **Owner:** ai-governance.

**C-12 · Excessive agency and the autonomy policy** — Priority **7/10** — *`S5` `S6`*
- **Probe:** Split the question three ways, as the taxonomy does: excessive *functionality* (tools it does not need), excessive *permissions* (scopes it does not need), excessive *autonomy* (decisions it should not make alone). **Audit each separately — a system can pass two and fail catastrophically on the third.** Then note what "alone" now means: there is nobody else. Every decision this system makes, it makes alone.
- **Passes only if:** least agency is enforced by default, autonomy is bounded per capability by **reversibility, deterministic validation and an infrastructure-enforced blast-radius cap** rather than by approval, and there is a kill switch that has been tested and that fires itself.
- **Targets:** a documented autonomy level per agent; least-privilege tool scopes with **read and write separated**; **irreversible and high-impact actions removed, made reversible, or gated by a deterministic validator with a blast-radius cap** (S5, S6) — "human approval" is not an available control in this system and must not be recorded as one; a kill switch that has been **tested, not merely present**, and that can be tripped automatically; a **scheduled, automated** permission review that actually removes unused scopes, because a review that depends on someone remembering will not happen.
- **On failure:** remember the compounding relation: **a prompt injection is only as dangerous as the permissions sitting behind it.** Cutting permissions is the cheapest risk reduction available to you anywhere in this document — and with the human backstop gone, it is also the most load-bearing.
- **Structural fix (S13):** The tool gateway (see `A-11`) — scopes are enforced at the door, and revoking one is a table edit rather than a code change in every agent.
- **Standing control:** *Unused scopes revoked automatically on a schedule; the kill switch tested.* Read and write scopes stay separated, asserted by test. **Ratchet (S11):** granted scope count monotonically decreasing absent a decision record — **permission-cutting is the cheapest risk reduction in this document and the only one that keeps paying after you leave.** **Blocks:** deploy on scope expansion without a record. **Owner:** platform-security.

**C-13 · AI management system** — Priority **6/10**
- **Probe:** Determine whether an AI management system is claimed. If it is, ask for the statement of applicability and the audit evidence per control. If it is not claimed, note that its absence is rarely fatal — but its presence shortens external diligence considerably.
- **Passes only if:** a functioning management system exists with a risk-based statement of applicability, and ideally third-party certification.
- **Targets:** a statement of applicability covering all 38 Annex A controls of the certifiable AI management standard (most deployments mark 30–36 applicable; the data controls relating to self-training may legitimately be not-applicable where no self-training occurs — but say so explicitly); an AI policy with defined roles; impact assessments; controls over third-party and model-provider dependencies; audit evidence for every included control, **emitted by the pipeline rather than collected by hand.**
- **On failure:** if you claim the management system, produce the statement of applicability. If you cannot, **drop the claim** — an unsubstantiated certification claim is worse than none.
- **Standing control:** *Every included control maps to an evidence artifact emitted by the pipeline.* A control whose evidence has gone stale opens a finding automatically. **Ratchet (S11):** controls with stale evidence remain at zero. **Blocks:** nothing directly; the certification claim is withdrawn if evidence lapses. **Owner:** ai-governance.

**C-14 · Validating the judge** — Priority **6/10** — **UNSETTLED DOCTRINE** — *`S2` `S3` — and the first move is not to validate the judge but to need it less*
- **Probe:** Find every place a model grades another model's output and gates a decision on it. For each, ask three questions in this order. **(1) Does this actually need judgement at all?** Most "judged" properties are verifiable — the citation resolves or it does not, the SQL runs or it does not, the JSON validates or it does not, the claim appears in the retrieved passage or it does not. **Every judged gate you can convert into a deterministic check is one less place where the machine assesses the machine — and in a system with no human reviewer, that closed assessment loop is the central structural risk of the entire architecture.** (2) If judgement is genuinely required, was the judge validated against labels that did not come from a model? (3) Is the judge a different model, from a different vendor, than the one under test? Strong judges do reach over 80% agreement with human raters — the same level humans reach with each other — **but they carry documented position bias, verbosity bias, and self-preference bias, with frontier models preferring their own outputs on the order of +10% to +25%.** An unvalidated judge silently launders that bias into every metric it gates, and every downstream decision inherits it.
- **Passes only if:** every judged gate that could have been deterministic **has been made deterministic**; and any judge that still gates a decision has been validated against an **externally-sourced, frozen label set** on a held-out split, with bias controls applied and re-validation on a schedule.
- **Targets:** an explicit register of judged gates, each with a written reason why it could not be made deterministic; judge-versus-label agreement measured on a held-out labelled set (target ≥0.8 agreement, or a defined kappa), where the labels come from real outcomes, a deterministic oracle, or a **human-labelled asset acquired once, out of band, and frozen** — legitimate precisely *because* it is a fixed artifact and not an in-loop review step; position-swap and verbosity controls applied; **the judge model is neither the model under test nor from the same vendor** (S2); periodic re-validation; the judge prompt and rubric under version control like any other behaviour-defining artifact.
- **On failure:** stop gating on it until it is validated — **and before you validate it, try to eliminate it.** And note the reflexive point that applies to you, the auditor, right now: this is exactly why Rules 6 and 7 of §1 exist, and why S2 exists at all.
- **Standing control:** *Judges expire.* Every judge that gates a decision carries a scheduled re-validation and **stops gating when its validation lapses — fail closed, not fail stale.** The judged-gate register is checked in; the judge model is asserted to differ from the model under test on every run, and re-asserted whenever either model is swapped. **Ratchet (S11):** the number of judged gates may only decrease; judge-label agreement may not regress. **Blocks:** release when a judge is unvalidated. **Owner:** ai-quality.

**C-15 · Guardrails have their own tests, and their own adversary** — Priority **7/10** — *`S2` `S3`*
- **Probe:** Guardrails are artifacts and need adversarial tests of their own. Measure their false-positive and false-negative rates **on your own traffic**. Treat any vendor claim of the "catches 95% of attacks" variety as unverifiable marketing — it was not measured on your data, your prompts, or your threat model.
- **Passes only if:** guardrails have a dedicated adversarial test suite with measured false-positive and false-negative rates and a CI gate on regression — **and the adversary is continuous, not calendared.**
- **Targets:** an adversarial suite (jailbreak plus injection) running in CI; measured FP and FN rates against agreed thresholds; a regression gate on **any** guardrail change; **a continuous automated red-team campaign — a harness that runs on a schedule, pulls fresh attack corpora, and reports attack-success rate as a tracked metric that regresses like any other.** A quarterly human red-team exercise is not available in this operating model, and it was always too slow anyway: **your attackers are automated, so your red team must be too.** Where an external human red-team engagement is procured, it is a *periodic assurance sample*, never the control. Results retained as evidence.
- **On failure:** measure before you trust, then automate the measuring. **A guardrail with an unmeasured false-negative rate provides an unknown amount of protection, which is operationally the same as none — except that people rely on it.**
- **Standing control:** *The adversarial campaign is continuous and its corpus is refreshed.* The red-team harness runs on a schedule, pulls fresh attack corpora, and reports attack-success rate as a tracked metric that regresses like any other. False-positive and false-negative rates are measured continuously on production traffic. **Ratchet (S11):** ASR may not rise; FN rate may not rise. **Where an external human red-team engagement is procured, it is a periodic assurance sample and is never counted as the standing control.** **Blocks:** release on regression. **Owner:** ai-security.

**C-16 · Machine identity and agent-to-agent trust** — Priority **7/10** — *`S6` `S9`*
- **Probe:** Inventory every non-human identity. Check whether any agent operates by **borrowing a human's session or sharing a service account** — if so, every action it takes is indistinguishable from that human's, in both the audit log and the blast radius. **In this system that is not merely untidy; it is the destruction of the only accountability mechanism you have left** (C-37), because provenance is the whole story once nobody signs anything.
- **Passes only if:** every non-human identity is inventoried, least-privilege, short-lived and auditable, and **no agent borrows a human's session.**
- **Targets:** a maintained machine-identity inventory; a managed identity per agent; scoped, expiring credentials; secrets in a vault; authenticated agent-to-agent calls; **scheduled, automated access reviews that decommission stale identities** rather than reviews that depend on someone running them. Machine identities now outnumber human ones by roughly two orders of magnitude, and a large share of the new ones are agents — **the identity plane is where this system's real attack surface now lives, and here it is essentially the only plane there is.**
- **On failure:** issue per-agent identities before you do anything else in this cluster. Everything downstream depends on being able to attribute an action to something.
- **Structural fix (S13):** Generate the identity inventory from the identity provider (§6.5.3); agents get their own identities from the broker (`B-25`). Borrowing a human session becomes structurally impossible rather than merely forbidden.
- **Standing control:** *The identity inventory is generated from the identity provider, not maintained by hand.* Stale identities are decommissioned automatically on a schedule, and a continuous assertion flags any agent action attributable to a human identity. **Ratchet (S11):** borrowed sessions and shared service accounts remain at zero — **provenance is the whole accountability story once nobody signs anything (C-37), and a borrowed identity destroys it.** **Blocks:** deploy. **Owner:** platform-security.

**C-17 · Tool poisoning** — Priority **7/10** — *`S1` `S9`*
- **Probe:** Read every tool definition and description the model is given — including the ones from third-party servers. **A malicious instruction hidden in a *tool description* is read by the model on every single invocation.** The canonical demonstration exfiltrated a user's SSH key through a poisoned "calculator" tool description; an analysis of 1,899 public connector servers found roughly 5.5% vulnerable to exactly this.
- **Passes only if:** tool definitions are trusted, pinned, and **re-verified at runtime** — not merely trusted at first connection.
- **Targets:** an allowlist of vetted tools and servers; **tool descriptions pinned on first use and diffed automatically on every change, with the change blocking until the diff passes a gate** — this pin-and-diff is the exact machine substitute for the engineer who would have noticed that a tool description changed overnight, and it is one of the cleanest substitutions in this document; runtime scanning of tool definitions; tool *outputs* treated as untrusted input (they are); an audit log of every tool invocation with its arguments.
- **On failure:** pin and diff. Unlike a single-session injection, **a poisoned tool definition persists across *every* session that uses that tool** — the compromise outlives its introduction indefinitely, and nothing in this system re-reads a tool description with suspicion.
- **Structural fix (S13):** The tool gateway (see `A-11`) — descriptions are pinned and diffed **at the door**, so a changed description is caught once, centrally, before any agent has read it. Without the door, this check must run in every place a tool is connected, forever.
- **Standing control:** *Tool descriptions pinned and diffed on every connect and on a schedule; a changed description blocks until the diff passes a gate.* **This is the highest-value standing control in Track C, because the change happens on somebody else's schedule and arrives without a pull request.** **Ratchet (S11):** unpinned tool definitions remain at zero. **Calibration (S12):** a modified tool description is injected monthly and must be caught before the tool is used. **Blocks:** tool use. **Owner:** ai-security.

**C-18 · Connector and tool-server security** — Priority **7/10** — *`S1` `S6`*
- **Probe:** For every external tool server the system connects to: is it authenticated? Is authorisation enforced per method, or only at connect? Can it be swapped for a different server under the same name? Are credentials passed inside tool arguments (they must never be)?
- **Passes only if:** only authenticated, authorised servers can connect, with least-privilege scopes and a complete audit trail.
- **Targets:** authentication and authorisation enforced **at the method level** (listing tools, calling a tool, reading a resource), not just at the connection; a server allowlist **that no agent identity can write to**; scope expiry; **no credentials in tool arguments**; an immutable audit trail; **automated scanning for tool shadowing and rug-pulls on every server update.** The first malicious connector package was identified in September 2025; a private-repository exfiltration through a legitimate connector is now the standard worked example of the agentic supply chain.
- **On failure:** allowlist and pin. **An unpinned third-party tool server is an unreviewed dependency with a live network connection into your agent's instruction context** — and in this system, *unreviewed* is not a qualifier. It is the baseline.
- **Standing control:** *Method-level authorisation asserted by test; shadowing and rug-pull scanning on every server update and on a schedule.* The server allowlist lives where no agent identity can write to it. **Ratchet (S11):** unauthenticated servers and credentials-in-arguments remain at zero. **Blocks:** connection. **Owner:** ai-security.

**C-19 · Memory and context poisoning** — Priority **6/10** — *`S9`*
- **Probe:** Determine what can write to persistent agent memory and to the retrieved context, and whether anything validates it. **Then attempt a plant: write content into memory in one session and see whether a *different* session acts on it.**
- **Passes only if:** persistent memory and retrieved context are integrity-controlled, cannot be silently tampered with, and poisoned entries can be **found and purged**.
- **Targets:** provenance and integrity checks on every memory write; memory-write auditing; validation of retrieved context before use; segregation of untrusted content from trusted memory; a **demonstrated** ability to purge or quarantine poisoned entries.
- **On failure:** the compromise here **outlives the session that introduced it** and can reshape behaviour across users and across runs — and it is far harder to detect after the fact than a live injection. Build the provenance trail now; you cannot retrofit it after the event. **This is the one attack class that survives every restart, every rollback and every redeploy in a system that has no long-term human memory of its own.**
- **Standing control:** *Provenance and integrity enforced at write time, not audited afterwards.* Memory writes are validated and logged; a quarantine-and-purge drill on a cadence proves poisoned entries can still be found and removed. **Ratchet (S11):** memory writes without provenance remain at zero. **Blocks:** the write. **Owner:** ai-platform.

**C-20 · Responsible-AI dimensions with owners and numbers** — Priority **6/10** — *`S3`*
- **Probe:** For each dimension that applies — fairness and bias, explainability, transparency, contestability, veracity, robustness, controllability — find the owning role, the concrete measure, and the monitoring cadence. **A policy statement with none of these three is decoration**, and decoration is what this system produces most fluently.
- **Passes only if:** each applicable dimension has an owning role, **an automated measure that runs on a schedule**, and a threshold that does something when crossed.
- **Targets:** bias and fairness testing on representative *and* edge-case sets, **running in CI**; an explainability and contestability path for affected users (contestability matters more here, not less: when a decision is made by a system nobody reviewed, the affected person's route to challenge it is the only correction mechanism left); robustness tests; a documented owning role per dimension; drift and refusal rates monitored in production with alerts. This is increasingly a scored section in enterprise procurement, and it is the framing a regulator will use when asking whether harm was foreseeable.
- **On failure:** assign owners, pick measures, and automate them. **"We take fairness seriously" is not a measure. Neither is a dashboard nobody looks at.**
- **Standing control:** *Each dimension's measure runs on a schedule with a threshold that alerts and blocks.* A dimension with no measurement in the current period opens a finding. Contestability is asserted by test — **when a decision is made by a system nobody reviewed, the affected person's route to challenge it is the last correction mechanism in the architecture, and it must actually work.** **Ratchet (S11):** dimensions with no running measure remain at zero. **Blocks:** release on a bias or robustness regression. **Owner:** the owning role per dimension.

**C-21 · Training and fine-tuning data governance** — Priority **7/10** — *(NOT-APPLICABLE only if no custom or fine-tuned model exists — say so explicitly)*
- **Probe:** For every dataset used to train or tune anything: what is its source, its licence, and its lawful basis? Can you trace it end to end? **If you cannot establish what went into a model, you cannot defend what comes out of it.**
- **Passes only if:** every training and tuning dataset has documented provenance, an explicit licence, and a lawful basis, traceable end to end.
- **Targets:** a dataset inventory with source, licence and lawful basis per entry; data-lineage records; exclusion of unlicensed or copyleft-incompatible data; **automated poisoning checks on ingested data**; the relevant data controls evidenced. This is simultaneously a legal exposure and a poisoning risk — compromised open adapters are a demonstrated attack, and an adapter is precisely the kind of dependency an autonomous system will pull in because it scored well on a benchmark.
- **On failure:** build the inventory. **In diligence, an unaccountable training set is treated as a contingent liability of unknown size** — and "unknown" is priced pessimistically.
- **Standing control:** *Governance enforced at ingest.* A dataset without a recorded source, licence and lawful basis cannot enter the training or tuning pipeline; poisoning checks are automated and re-run on every retrain. **Ratchet (S11):** unaccounted datasets remain at zero. **Blocks:** ingest and retrain. **Owner:** ai-governance + legal.

**C-22 · Retrieval evaluated separately from generation** — Priority **7/10** — *`S3`*
- **Probe:** Take ten wrong answers from the system. For each, determine whether the retriever failed to find the right context, or the generator ignored the context it was given. **If you cannot tell them apart, you cannot fix either one** — and every improvement attempt will be a guess. In a system where the improver is also a machine, guessing compounds.
- **Passes only if:** retrieval and generation are evaluated independently, with thresholds that **localise a failure to a specific component.**
- **Targets:** retriever metrics (context precision and recall) and generator metrics (faithfulness and answer relevancy) measured **separately** in CI; a faithfulness floor — commonly around 0.85, since fabrication is the worst failure mode this system has; a labelled evaluation set that supports recall measurement; CI exiting non-zero below the floor; a written retriever-versus-generator diagnosis for each recorded failure. **This decomposition is what allows a machine to localise a fault without a human to reason about it — it is the substitution for engineering intuition, and it is a good one.**
- **On failure:** decompose the metric first. **This single change turns "the AI is sometimes wrong" into an actionable engineering problem** — and an actionable problem is one an agent can fix without supervision.
- **Standing control:** *Retriever and generator measured separately in CI, blocking below the floor.* The labelled set is refreshed on a cadence. **Ratchet (S11):** faithfulness may not regress below its floor; the floor may rise. **Blocks:** merge and release. **Owner:** ai-quality.

**C-23 · Personal data in prompts, logs and traces** — Priority **8/10** — *`S3`*
- **Probe:** Grep the log store, the trace store and the prompt-capture store for email addresses, phone numbers, national identifiers and card patterns. **You will very likely find them.** Prompt and completion logging routinely captures personal data into stores that sit outside the data map, outside the impact assessment, and outside the retention policy — it is one of the most common and most avoidable findings in this entire field, and it happens precisely because nobody looked at what the logger was logging.
- **Passes only if:** prompt and completion capture is governed, minimised and access-controlled exactly like any other store of personal data — **because that is what it is.**
- **Targets:** redaction or tokenisation **before** logging, enforced at the emitter rather than by a downstream cleanup job; explicit retention limits on prompt and completion logs; the observability store access-controlled and audited; the privacy impact assessment explicitly covering logging; user opt-outs honoured downstream in **every derived store**; **no secrets and no personal data in traces, verified by an automated scan that runs continuously and blocks the build** — not by an occasional inspection.
- **On failure:** redact at the edge, then go back and purge what has already been captured. And check the retention: **an observability store with infinite retention is an infinite liability.**
- **Structural fix (S13):** The typed telemetry emitter (see `B-07`): redaction happens at the emitter, structured fields only, no free-text log line carrying model input. **A new logging sink cannot bypass a redactor it must pass through in order to exist** — which closes the exact recurrence path named in this check's standing control.
- **Standing control:** *Redaction enforced at the emitter; the stores scanned continuously.* An automated personal-data scan of the log, trace and prompt stores runs on a schedule and blocks the build on a hit. **A new logging sink must register with the scanner, enforced by a registry check — that registration requirement is the recurrence path, and it is where this finding comes back.** **Ratchet (S11):** personal data in traces remains at zero; retention windows may shorten, not lengthen. **Blocks:** merge and deploy. **Owner:** privacy + platform-observability.

**C-24 · Disclosure through memorisation and system-prompt leakage** — Priority **7/10** — *`S1` `S3`*
- **Probe:** Ask the model for its system prompt in ten ways. Attempt extraction and regurgitation attacks. Then check what is *in* the system prompt — **if the answer includes an API key, a database identifier, an internal URL, or a rule that is load-bearing for security, you have found a serious defect.** The blunt rule: **system prompts are not security controls.** A divergence attack on an aligned production model has been shown to induce training-data emission at roughly 150× the normal rate, extracting megabytes for about $200 of queries.
- **Passes only if:** no secrets live in prompts or system context, and memorisation/extraction risk is tested and bounded rather than assumed away.
- **Targets:** no credentials and no personal data in system prompts, **enforced by an automated scan in CI that blocks**; **all security controls enforced deterministically outside the model** — which in this operating model is not merely good practice but the founding principle: a rule the model has been *asked* to follow is a preference, and preferences do not survive an adversary; extraction and regurgitation tests in the evaluation suite; output filtering for sensitive patterns; a defined incident path for a disclosure event.
- **On failure:** move every control out of the prompt and into code. **A rule the model has been asked to follow is a preference; a rule the code enforces is a control.** There is nobody standing behind the preference.
- **Structural fix (S13):** The emitter (see `B-07`) plus the prompt registry (see `A-20`) — no code path exists that can put a credential into a system prompt, and none exists that can log one.
- **Standing control:** *Automated scan asserts no credentials or personal data in system prompts, on every build.* Extraction and regurgitation tests run in the evaluation suite on every model change. **Ratchet (S11):** secrets in prompts remain at zero; extraction success may not rise. **Blocks:** merge and release. **Owner:** ai-security.

**C-25 · Copyright and licensing of generated code** — Priority **6/10** — *`S1` `S9`*
- **Probe:** Scan the generated code for copyleft-incompatible fragments and reproduced snippets. Then find the written position: **what does this organisation believe it owns, and on what basis?**
- **Passes only if:** AI-generated code is licence-scanned and IP-cleared **by a blocking gate** before merge, and the organisation's position on the intellectual-property status of model output is written down.
- **Targets:** licence and composition scanning of generated code including copyleft detection; a provenance record per generated artifact (S9); a policy on which assistant tools are acceptable and under what terms; a documented IP position on model output; **the scan itself as the merge gate (which is A-01 again, from a different direction) — because there is no reviewer to notice a copied snippet, the scanner is not an assistant to the control. It *is* the control.** And note the consequence that is unusually sharp in a system like this one: purely machine-generated output lacks the human authorship that copyright requires in the US, and here **essentially all of it is purely machine-generated.**
- **On failure:** in an acquisition, unclear provenance on machine-generated code is a live valuation and indemnity issue — **this is a legal exposure wearing an engineering costume**, and the no-review operating model makes the exposure larger, not smaller, because it removes the last human fingerprints from the artifact.
- **Standing control:** *Blocking licence scan on every build; snippet-filter bypass rate monitored.* **Cadence:** the written IP position is reviewed on a fixed schedule against legal developments — in-command, calendared, with an owning role. **Ratchet (S11):** unresolved copyleft conflicts remain at zero. **Blocks:** merge. **Owner:** legal + platform-security.

**C-26 · SBOM, AI-BOM, provenance and cyber-resilience obligations** — Priority **8/10** — *`S1` `S9`*
- **Probe:** Take a released artifact. Is there an SBOM? Is there an AI-BOM covering the models, datasets and adapters? Is the provenance signed? And critically: **is any of it verified at deploy, or merely archived?**
- **Passes only if:** every release artifact ships with an SBOM and signed provenance at a target build level, and that provenance is **verified at deploy** — enforced, not filed.
- **Targets:** an SBOM (SPDX or CycloneDX) per build; an AI-BOM for the AI components; hardened build provenance (Build Level 3) on every externally distributed artifact, Level 2 minimum internally; **provenance verified at deploy by an admission policy that fails closed**; documented regulatory-readiness evidence. The EU cyber-resilience regime entered into force in December 2024, with reporting obligations from September 2026 and full application from 11 December 2027; exposure up to €15M or 2.5% of global turnover.
- **On failure:** **generating an SBOM nobody checks satisfies a checkbox and prevents nothing. Wire the verification** — and note that in this system, *nobody checks* is not a lapse. It is the specification.
- **Standing control:** *SBOM and AI-BOM generated per build; provenance verified at deploy by a policy that fails closed.* **Calibration (S12):** an unsigned artifact is offered for deployment on a schedule and must be refused — **verification that is never exercised is verification you are trusting, and in this system nobody checks anything by hand.** **Ratchet (S11):** unverified deploys remain at zero. **Blocks:** deploy. **Owner:** platform-security.

**C-27 · Sector compliance baseline** — Priority **9/10** — *audit only the frameworks that genuinely bind this product; determine applicability first*
- **Probe:** Establish which frameworks actually apply, then demand in-date evidence for each. Check for lapsed controls and expired attestations — **a certificate that expired six months ago is not evidence, it is a finding.**
- **Passes only if:** every applicable framework is currently certified or attested, with in-date evidence and no lapsed controls.
- **Targets:** a current SOC 2 Type II report and/or ISO 27001 certificate where relevant; if payment data is handled, PCI DSS v4.0.1 with **all** of the 51 future-dated requirements implemented (they became mandatory on 31 March 2025 — including the client-side payment-page controls — and there is no grace period left); healthcare agreements and safeguards where health data is processed; **every control's evidence dated, owned and traceable to a specific system, and emitted continuously by the pipeline rather than assembled the week before the audit.** General security certification carries across but never substitutes for sector-specific obligations. Note also that several of these frameworks assume a change-management process with human approval; where they do, you must be able to demonstrate that the deterministic gate (A-01, B-35) satisfies the *control objective* — segregation of duties, traceability, authorised change — and not merely that you have automated something. **Bring that argument to the assessor deliberately. Do not let them discover it.**
- **On failure:** scope first, evidence second. Auditing against a framework nobody is subject to wastes the engagement.
- **Standing control:** *Evidence emitted continuously and dated; expiries alert before they lapse.* An attestation or certificate approaching expiry raises a finding automatically ahead of the date; control coverage is re-mapped when a framework publishes a new version. **Ratchet (S11):** lapsed controls remain at zero. **Cadence:** the argument that the deterministic gate satisfies the change-management control objective (segregation of duties, traceability, authorised change) is maintained as a living artifact and taken to the assessor deliberately — **do not let them discover the absence of human approval; bring it to them with the evidence.** **Blocks:** release on a lapsed control. **Owner:** compliance.

**C-28 · Residency and regulatory scope decided at design time** — Priority **7/10**
- **Probe:** Where does the data physically live? **Include the parts people forget: the vector store, the caches, the logs, the traces, the queue, the backups, and the model provider's inference region.** Then ask when this was decided — and whether it was decided at all, or merely defaulted to by a model choosing a region.
- **Passes only if:** residency and regulatory scope are architectural decisions taken at design time and **enforced by the platform, not by convention** — because convention requires someone to uphold it.
- **Targets:** a documented data-residency architecture per region; regulatory scope determined at design review; region-pinned storage and processing **including vector stores, caches and logs**, asserted by an automated check on every deploy; controls mapped to obligations before the build started.
- **On failure:** flag this early and loudly. **Re-regionalising a vector store or a log pipeline after launch is a re-architecture, not a configuration change** — it is cheap to design in and brutally expensive to retrofit, and reviewers know it.
- **Structural fix (S13):** The derived-store registry (see `B-37`) — region is a required property of registration, so an out-of-region store cannot be constructed.
- **Standing control:** *Region-pinning asserted on every deploy across storage, cache, log, trace, queue, backup and inference region.* A new store in the wrong region fails the deploy. **Ratchet (S11):** out-of-region stores remain at zero. **Blocks:** deploy. **Owner:** platform-infrastructure.

**C-29 · Content safety and misuse prevention** — Priority **6/10** — *`S3` `S8`*
- **Probe:** Find the content-safety policy and its taxonomy. Then test the classifiers and measure their false-positive and false-negative rates. Then check the escalation path — **and confirm it has been exercised at least once, rather than merely written.** Ask specifically what the escalation *does*, because an escalation that ends at a person's inbox ends nowhere in this system.
- **Passes only if:** a documented content-safety policy is enforced by tested guardrails, misuse is monitored in production, and the escalation path **executes** rather than notifies.
- **Targets:** a content-safety policy and taxonomy; input and output classifiers with **measured** FP/FN rates; abuse monitoring and rate limits **with automatic throttling and suspension**, not alerting; prohibited-content safeguards on generative features — note the new EU prohibition covering non-consensual intimate imagery capability takes effect 2 December 2026, **with a safe harbour for operators who can demonstrate effective preventive safeguards, which means the evidence you build now is the defence later**; **an escalation runbook that executes automatically, with an out-of-band human path as break-glass only** (S10).
- **On failure:** write the policy first — you cannot classify against a taxonomy that does not exist. Then automate the response, because a safety control that waits for a person is a safety control that is off at night.
- **Standing control:** *Classifiers measured continuously; abuse monitoring throttles automatically; the escalation runbook executes.* The runbook is exercised on a schedule. **Ratchet (S11):** false-negative rate may not rise. **The prohibited-content safeguards evidence is retained continuously, because it is the safe-harbour defence and it must exist before the event, not after.** **Blocks:** release on regression. **Owner:** trust-and-safety.

**C-30 · Jailbreak resistance** — Priority **6/10** — **UNSETTLED DOCTRINE** — *`S8`*
- **Probe:** Measure attack-success rate against a standard benchmark. **Resistance here is probabilistic, never absolute** — and the measurement method matters as much as the number, since earlier evaluators systematically over-estimated attack success. **Report the method alongside the figure or the figure is meaningless.**
- **Passes only if:** jailbreak resistance is measured against a standard benchmark, with a tracked attack-success-rate ceiling enforced per release, and residual risk documented with a compensating control and a runtime tripwire.
- **Targets:** attack-success rate measured on a recognised benchmark; a defined ceiling **enforced as a release gate** (S1), not merely tracked; **layered defence rather than a single classifier**; a regression gate; **residual risk written down against an owning role, with a compensating control and an executable tripwire that fires if the risk materialises in production** — a signature was never the control here, and it is not available in any case; the ceiling re-measured on every model change, because a provider-side model update can move it without you touching anything (B-13, B-24).
- **On failure:** measure, do not assume. And **do not report a jailbreak-resistance number without the benchmark and method that produced it** — an uncontextualised number in a data room is a liability.
- **Standing control:** *Attack-success rate measured on every model change and on a schedule; the ceiling enforced as a release gate.* **Ratchet (S11):** the ASR ceiling may tighten, never loosen. Residual risk is re-accepted on a cadence and its tripwire verified. **A provider-side model update can move this number without you touching anything (B-13, B-24), so the measurement is bound to the model version, not to the calendar alone.** **Blocks:** release. **Owner:** ai-security.

**C-31 · Adversarial technique taxonomy and agentic threat layers** — Priority **5/10**
- **Probe:** Map the AI threat model against a living adversarial-technique knowledge base, and decompose the agentic system across the seven-layer agentic threat-modelling structure (foundation models, data operations, agent frameworks, deployment infrastructure, evaluation and observability, security and compliance, the agent ecosystem). **The purpose is coverage: gaps become visible when you are forced to name the layer.**
- **Passes only if:** the AI threat model references specific adversarial techniques explicitly, and the agentic system is decomposed by layer so that coverage gaps are visible.
- **Targets:** the threat model mapped to named tactics and techniques — **cite live figures at time of use, since the knowledge base is continuously updated**; the agentic system decomposed across all seven layers; each identified technique mapped to a control or a test; **the mapping refreshed automatically as the taxonomy updates**, because a taxonomy that must be manually re-read will be re-read once.
- **On failure:** useful as shared vocabulary and as a coverage check; rarely a standalone blocker. Do it after the higher bands.
- **Standing control:** *The mapping refreshes automatically when the taxonomy updates.* New techniques without a mapped control or test become findings. **Ratchet (S11):** unmapped techniques may only decrease. **Blocks:** nothing by default. **Owner:** ai-security.

**C-32 · Vector and embedding weaknesses** — Priority **6/10** — *`S6`*
- **Probe:** Check access control and tenant isolation on the vector store. **Then internalise the finding that changes how you must classify it: text embeddings can be inverted back into text.** Published work recovers 92% of 32-token inputs *exactly* and reconstructs full names from clinical notes. The operational consequence is unambiguous — **a vector-store breach is a plaintext breach, and must be classified, reported and remediated as one.**
- **Passes only if:** vector stores are access-controlled, residency-compliant and hardened against inversion and corpus poisoning — **and treated as personal-data stores wherever they embed personal data.**
- **Targets:** access control and tenant isolation on the vector store; region-pinned for residency (see C-28); inversion mitigations where embeddings encode sensitive text; corpus-poisoning checks on ingested content; retrieval-integrity monitoring.
- **On failure:** **re-classify the store first** — that single reclassification pulls it into the data map, the impact assessment, the retention policy and the breach-notification path, where it always belonged. Nothing else in this document is so cheap to fix and so expensive to have got wrong.
- **Structural fix (S13):** Register the vector store through the derived-store registry (see `B-37`). **That one registration makes it inherit retention, erasure and breach handling automatically and permanently** — worth more than any control bolted on afterwards.
- **Standing control:** *The vector store is registered as a personal-data store in the data map.* That single registration makes it inherit C-04's retention, erasure and breach-notification paths **automatically and forever**, which is worth more than any control you could bolt on afterwards. Access control and residency are asserted on every deploy; corpus-poisoning checks run continuously. **Ratchet (S11):** unregistered embedding stores remain at zero. **Blocks:** deploy. **Owner:** privacy + ai-platform.

**C-33 · An AI usage policy that is enforced, not merely published** — Priority **6/10** — *`S1`*
- **Probe:** Find the policy. Then find evidence anyone — **or anything** — follows it. **Publication is not adoption, and in this operating model the primary audience for the policy is not a person at all.** So ask the version of the question that actually bites: which clauses of this policy are enforced by the platform, and which are merely written?
- **Passes only if:** a clear, communicated, cross-functionally owned AI usage policy exists, is current, **and every machine-relevant clause is mechanically enforced** rather than trusted.
- **Targets:** a published policy covering expectations, permitted tools, support for experimentation, and applicability by role; cross-functional ownership (engineering, legal, security, IT, product); a living document reviewed on a fixed cadence; **permitted tools enforced by an allowlist, permitted models enforced at the gateway, permitted change classes enforced by the policy bundle** (S1) — **a policy that exists only as a document is, in a system with no reader, a policy that does not exist**; evidence of adoption, not just of publication. The evidence is clear that a communicated stance amplifies AI's positive effect, while ambiguity produces over-cautious and reckless behaviour *simultaneously* — the absence of a policy is itself a policy, and it is the worst one.
- **On failure:** write it, then **compile it.** Every clause you can move from prose into the policy bundle is a clause that will still be true next quarter.
- **Standing control:** *Every machine-relevant clause is compiled into the policy bundle and enforced.* Permitted tools are an allowlist, permitted models are enforced at the gateway, permitted change classes are enforced by the bundle. **A clause that cannot be compiled is flagged as unenforced inside the policy document itself, so the gap is visible rather than assumed away.** **Ratchet (S11):** unenforced machine-relevant clauses may only decrease. **Cadence:** reviewed on a fixed schedule with cross-functional owning roles. **Blocks:** merge on a policy violation. **Owner:** engineering-leadership + legal.

**C-34 · Provider training on your data — check the contract *and* the switch** — Priority **7/10** — *`S3`*
- **Probe:** Read the contract. Then check the actual configured state of the account. **These two frequently disagree**, and only one of them determines what actually happens to your customers' data.
- **Passes only if:** provider training on customer data is contractually disabled (or explicitly and lawfully consented to), configured accordingly, and **re-verified automatically on a schedule.**
- **Targets:** the provider contract confirms no training on prompts and completions, or documented user consent exists; the opt-out is **configured and evidenced** with an API assertion rather than a screenshot, so that it can be re-checked by a machine; a data-processing agreement in place; a data-flow diagram showing no unintended training feed; **an automated periodic assertion against the provider's API, plus monitoring for provider term updates** — the terms change, the change is not announced to you, and in this system there is nobody subscribed to the mailing list.
- **On failure:** configure it, assert it programmatically, schedule the re-check. **This is a one-hour task that closes a direct contractual and regulatory exposure** — and one of the few in this document that stays closed once you automate it.
- **Structural fix (S13):** The model gateway (see `B-13`) — the provider configuration is asserted at the door on every call, rather than screenshotted once into a folder.
- **Standing control:** *Automated API assertion of the opt-out state on a schedule, plus provider term-change monitoring.* A change in the asserted state opens a finding immediately. **The terms change, the change is not announced to you, and there is nobody subscribed to the mailing list** — so the assertion is programmatic and recurring, not a screenshot in a folder. **Ratchet (S11):** unasserted provider configurations remain at zero. **Blocks:** release on a state change. **Owner:** legal + ai-platform.

**C-35 · Benchmark contamination and who evaluates the evaluator** — Priority **4/10** — **UNSETTLED DOCTRINE** — *`S2`*
- **Probe:** Determine whether any evaluation set has leaked into training or been sent to a provider. Ask where the numbers in the pitch deck came from, and whether they would survive a hostile re-run. **Then note where you are standing: "who evaluates the evaluator" is not a footnote in this system. It is the central structural question of the entire operating model** — see A-39 and C-14 — and this check is where you write down the honest answer.
- **Passes only if:** evaluation integrity is protected against contamination and the evaluator itself is validated, so that reported numbers mean what they claim to mean.
- **Targets:** held-out private evaluation sets never exposed to training or to a provider; contamination checks whenever a public benchmark is quoted; the judge validated against an externally-sourced label set (C-14); documented provenance of the evaluation data (S9); periodic **automated** rotation of evaluation items, because a static held-out set held out by a machine that also writes the code will not stay held out for long.
- **On failure:** this matters most precisely when evaluation numbers are used in product claims, in marketing, or in a data room — **which is exactly the context you are operating in.** A number you cannot defend under a hostile re-run should not leave the building.
- **Structural fix (S13):** Hold the evaluation sets in a store the pipeline has no credential for (see `B-10`, `B-25`). **"Do not send the held-out set to the provider" is a rule an agent will break cheerfully and without malice. A missing credential is not a rule.**
- **Standing control:** *Held-out sets live where the pipeline cannot send them to a provider — enforced technically, not by policy.* Evaluation items rotate on a schedule; a contamination check runs whenever a public benchmark is quoted. **Ratchet (S11):** exposed evaluation sets remain at zero. **Blocks:** publication of a benchmark claim that fails the contamination check. **Owner:** ai-quality.

**C-36 · Transparency and marking of generated output** — Priority **6/10**
- **Probe:** Check whether users are told when they are looking at machine-generated content, and whether generative output carries machine-readable marking. Then check what is claimed about that marking.
- **Passes only if:** generated output is disclosed to users and machine-readably marked where legally required, **with the technical limits of marking documented rather than overclaimed.**
- **Targets:** clear disclosure in the user interface; content credentials plus machine-readable marking on generative output by the applicable deadline (2 December 2026 in the EU); a detection capability for ingested content; **a written acknowledgement that marking is a friction layer, not proof of provenance** — imperceptible watermarks are not adversarially robust and can be degraded by a determined attacker with detector access, and the leading credentials specification is explicit that absent credentials are not evidence of inauthenticity.
- **On failure:** disclose in the interface now (cheap, and it is the part users actually see); mark before the deadline; and **never claim the marking proves more than it does.**
- **Standing control:** *The disclosure is asserted by a UI test on every build; the marking is asserted on generative output by a test.* **Cadence:** the written statement of what marking can and cannot prove is reviewed on a schedule. **Ratchet (S11):** undisclosed generative surfaces remain at zero. **Blocks:** merge. **Owner:** product + legal.

**C-37 · Accountability without a signature** — Priority **5/10** — *`S9` — escalates to `BLOCKER-1` if any production component has no attested provenance chain*
- **Probe:** The classical question was *when the machine-written code causes harm, whose name is on it?* **In this system the honest answer is: nobody's, and nobody's ever will be.** Nobody signed the diff and nobody is going to. So accountability cannot be discharged by a signature, and pretending otherwise — collecting a sign-off from someone who did not read the change — is not accountability. It is a forged one, and it is worse than none, because it will be produced in court as though it meant something. **The real question is therefore: take any random line in production. Can you reconstruct it?** Which model produced it, from which prompt, against which frozen specification, gated by which policy bundle, passing which tests, attacked by which adversarial verifier and with what result, and which owning role is answerable for the component it lives in. **That chain is the accountability record.** If you cannot produce it, nobody is answerable — and the absence of a human signature makes that worse, not better.
- **Passes only if:** every production artifact carries an **attested provenance chain** — model, version, prompt, specification, policy bundle, test evidence, adversarial-verification result — and a **named owning role**, recorded in a registry, is answerable for the component.
- **Targets:** provenance tags on every commit (model, prompt hash, specification identifier, policy-bundle version); **the chain reconstructible for any line in production, on demand, by machine**; an owning role per component, recorded and current; **a written policy naming who is accountable for a defect in machine-authored code — and it must name a role, because it cannot name a reviewer**; an immutable audit trail of agent activity; provenance verified at deploy and not merely recorded (B-09, C-26).
- **On failure:** **build the chain.** The organisation is accountable whether or not anyone signed. The only question is whether it can *demonstrate* what happened — and a signature from a person who never read the diff was never that demonstration. **It was a comfort. The chain is the real thing.** AI amplifies whatever the organisation already is; it does not absorb its accountability, and removing the reviewer does not remove the liability. It only removes the alibi.
- **Standing control:** *The chain is verified at deploy and reconstructed on demand.* **A spot-reconstruction of a randomly chosen production line runs on a schedule and fails loudly when it cannot be completed** — that scheduled reconstruction is the accountability control, and it is the only thing that will notice the chain breaking. **Ratchet (S11):** production components without an attested provenance chain remain at zero (a breach is `BLOCKER-1`, §3). **Cadence:** the ownership registry is re-validated periodically — **roles change, and a registry naming a role that no longer exists is a registry naming nobody.** **Blocks:** deploy. **Owner:** engineering-leadership.

**C-38 · Fabrication and the downstream reliance on it** — Priority **6/10** — *`S3` — with no reviewer, the machine must refuse to present what it cannot verify*
- **Probe:** Take every citation, reference, link and identifier the system generates and **resolve it.** The ones that do not resolve were invented. Then find the places where a user acts on the output without any means of checking it — **that is where the real risk lives. The danger is not the error; it is the reliance.** And note the compounding: in this system there is no reviewer between the model's fabrication and the user's reliance. The user *is* the review layer, and they did not consent to the job.
- **Passes only if:** factual-accuracy risk is measured, and **high-stakes outputs are verified deterministically before a user can act on them** — with anything unverifiable withheld or clearly marked as unverified, never presented as fact.
- **Targets:** a groundedness/faithfulness evaluation (C-22); **citations verified programmatically against the retrieved sources rather than generated** — a citation that does not resolve is a **build error**, not a stylistic issue; claims cross-checked against the retrieval passage and numbers recomputed rather than restated; **where an output cannot be verified deterministically, it is withheld, degraded to a clearly-marked unverified state, or the feature is not shipped** — this is the substitution for the human reviewer of high-stakes output, and it is a strict improvement on one, because a machine refusing to present what it cannot verify never gets tired at 5pm on a Friday; confidence and uncertainty surfaced honestly in the interface; hallucination rate monitored in production against a defined threshold with an automatic degradation on breach.
- **On failure:** wire the citation check first. **It is mechanical, it is cheap, and it converts a class of invisible failure into a build error** — which is the only place failures in this system are ever visible.
- **Standing control:** *Citation resolution is a build error, continuously.* Unverifiable output is refused by a tested code path rather than presented with a caveat; hallucination rate is monitored with automatic degradation on breach. **Ratchet (S11):** unresolved citations remain at zero; hallucination rate may not rise. **Blocks:** merge and release; automatic degradation at runtime. **Owner:** ai-quality.

**C-39 · Lifecycle process and vocabulary standards** — Priority **4/10**
- **Probe:** Only material if an AI management system is claimed (C-13). If it is, the lifecycle processes, the risk-management method, the impact assessment and the terminology all need to align to the corresponding standards, or the claim is unsupported.
- **Passes only if:** lifecycle processes and terminology align to the standards suite wherever a management system is claimed — **so that the claim is substantiated rather than asserted.**
- **Targets:** lifecycle processes mapped to the AI lifecycle-process standard; risk management performed per the AI risk standard; impact assessment per the impact-assessment standard; terminology consistent with the vocabulary standard; all of it referenced from within the management-system documentation.
- **On failure:** align it, or drop the claim. Foundational scaffolding — rarely a standalone blocker, but **a hollow certification claim is an own goal in diligence.**
- **Standing control:** *Alignment re-checked when the management system is re-certified or a standard is revised.* **Ratchet (S11):** unsubstantiated claims remain at zero — drop the claim rather than carry it. **Blocks:** nothing by default. **Owner:** ai-governance.

**C-40 · Societal and environmental impact** — Priority **3/10**
- **Probe:** Assess whether the material impacts of this system — societal and environmental — are non-trivial. For most business web applications they are not; where they are, measure with a real methodology.
- **Passes only if:** material societal and environmental impacts are assessed, and the energy footprint of the AI workloads is measured with a real method where feasible.
- **Targets:** a carbon-intensity figure per functional unit (the ISO/IEC 21031:2024 method) or an equivalent estimate for the AI workloads; a societal impact assessment for high-impact use cases; efficiency measures documented; portfolio-level energy reporting where a regulatory duty applies.
- **On failure:** a context item, not a gate — unless sustainability reporting obligations bind you, in which case re-band it. **Document immateriality honestly rather than inventing a number.**
- **Standing control:** *Measured per release where material; materiality re-assessed when the workload changes shape.* **Ratchet (S11):** none required unless a reporting duty binds, in which case re-band. **Blocks:** nothing by default. **Owner:** platform.

## Structural remediation carried over from Part 1

Part 1 §6.5 identifies the doors that make whole classes of defect unrepresentable rather than merely detectable — **read it there for the method and the other tracks' rows.** Part 1 sized these doors with this volume's rooms already in view; the rows below are those rooms, repeated here for navigation:

| The door | What it makes impossible | Track C checks it collapses |
|---|---|---|
| Tenancy in the data-access layer | Writing the cross-tenant bug at all | `C-01` (`STOP-SHIP`) |
| One mediated tool gateway | Registering an unlabelled tool; an agent exceeding its scope; an irreversible action with no compensating path | `C-06` `C-08` `C-12` `C-17` |
| One model gateway | An unpinned model; an untagged cost; an unasserted provider setting | `C-34` |
| One typed telemetry emitter | A log line carrying raw model input; an interaction that cannot be replayed | `C-23` `C-24` |
| A derived-store registry | A store you cannot erase from; a store in the wrong region; a vector store nobody classified | `C-04` `C-28` `C-32` |

| A credential broker scoped to its own environment | A non-production workload minting a production credential | `C-16` |

**Evidence semantics for a shared door — the part that keeps the audit honest:**

- A door that **stands at this volume's Phase-0′ baseline** — built by Part 1, or pre-existing — may support a Track C check's initial verdict, but its applicability to the Track C case must itself be evidenced (the door has no gap specific to this case), and that evidence is recorded on the check.
- A door **built or widened during this volume's remediation** closes the corresponding finding **only after a fresh Phase-6′ re-run** of that check — and a widening triggers re-verification of every Part 1 check the door had already closed. It never erases, rewrites or upgrades a baseline verdict: the initial finding stays visible in the audit trail, and the structural ledger records `baseline_affected_checks`, `door_built_by_change`, `checks_requiring_fresh_rerun`, `initial_verdict_preserved: true` and `current_verification_reference` (Part 1 §6.5.6).
- The audit therefore always distinguishes **the baseline finding** from **the current state** — a root cause closed by a shared door is a cheap fix, never a retroactive pass.
- Where Part 1's structural pass chose to police a check rather than design it away (a recorded decision under Rule 13), the corresponding row above does not apply, and the check is remediated directly here.

Verifying that a standing door holds for a Track C case is real work — measured in hours, not days — and it is the dividend of Part 1 having sized its doors for these rooms. Where Part 1 undersized one, the widening is this volume's work, and the re-verification bill that comes with it is part of the sequence's price.
